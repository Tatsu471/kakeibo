# 🛠️ SakuToko：食費・交通費トレードオフ可視化計器 設計図

> **作成日**: 2026-03-25 (更新: 2026-03-27)
> **概要**: 食費と交通費のトレードオフを可視化し、未来の体験を「咲かせる」ためのログアプリ。伴走キャラクター「りりむ（Lilim）」が見守る。
> **ターゲット**: 個人利用（タツミさん自身）
> **動作環境**: Webブラウザ (Flutter Web想定)

---

## 1. プロダクト要件 (MVPフォーカス)

### 必須要件（Must: Day 1 プラン）
- [x] **Home画面**: 当月の食費・交通費合計のフラットな表示と、りりむの静的配置
- [x] **記録入力画面**: 手入力用の「電卓型キーパッド」と「食費/交通費」タブ切り替え。保存完了時のスナックバー通知
- [x] **振り返り画面**: 日付降順のシンプルな支出一覧リスト（ListView）と、思考を促すりりむの固定テキスト
- [ ] **認証**: 個人データを守るためのGoogle認証 (Firebase Auth)

### 開発スコープ外（MVPに含めない・やらないこと）
- [ ] 織璃無の表情差分やアニメーション、進捗に応じた状態変化
- [ ] 家賃、交際費、日用品など、食費と交通費以外の記録カテゴリ
- [ ] 予算超過時の警告機能や、支出に対するシステム側からのジャッジ（過剰な演出）

### 推奨要件（Want - Phase 2以降の拡張）
- [ ] **OCR機能 (Google ML Kit)**: 記録入力画面のカメラ起動。レシートから金額と日付を自動読み取り＆フォームへの自動代入
- [ ] **グラフ表示**: 振り返り画面での月間の支出推移を可視化する棒グラフ（`fl_chart` パッケージ等を使用）

## 2. 画面およびUI詳細設計

### 1. トップ画面（ホーム画面）
現在の支出合計をひと目で把握できる一覧画面。
- **レイアウト構造**: 縦並び（Column）をベース。
- **上部要素**: 当月の「食費合計額」を大きなフォントで表示。
- **中央要素**: 当月の「交通費合計額」を表示。
- **キャラクター配置**: 食費と交通費の間に生じる余白へ、りりむ（Lilim）のSDキャラクター画像を配置。
- **下部要素**: ボトムナビゲーションバー、または横並び（Row）のアイコン。左から順に「設定」「カメラ（OCR起動）」「手動入力（＋）」「グラフ・履歴」。

### 2. 記録入力画面
直感的に素早く記録を完了させるためのシンプルなフォーム画面。
- **レイアウト構造**: タブ切り替えと入力フォームの組み合わせ。
- **タブ（TabBar）**: 上部で「食費」と「交通費」を切り替え。
- **金額フォーム**: 数値入力用のキーパッド（標準数値キーボード、または画面内の独自電卓UI）。
- **メモ・種類フォーム**: 任意のテキスト入力フィールド。
- **OCR連携 (Want)**: カメラアイコン押下でカメラ起動、解析結果を「日付」「合計」に自動代入。
- **保存処理**: 決定ボタンでFirestoreへ書き込み。完了後は**スナックバー**で通知しトップ画面へ戻る。

### 3. 振り返り画面（グラフ・履歴）
記録したデータを比較し、自身の行動を分析するための画面。
- **データ取得**: Firestoreから当月のデータを日次で取得し、状態管理パッケージで保持。
- **履歴リスト**: 日付降順で支出データをリスト表示（ListView）。
- **振り返りの問い**: 画面の一部に織璃無のアイコンと吹き出し。「過去の記録だね。どう感じる？」といった、ユーザーに思考を促すフラットなテキストを固定表示。

### 非機能UX要件（トーン＆マナー）
- グラスモーフィズムを採用し、全体を淡いトーンで統一。
- フォントには丸みのある手書き風フォントを採用。織璃無のテーマカラーをアクセントに使用。

## 3. アーキテクチャと技術スタック

- **Frontend**: Flutter Web (状態管理の導入やUI構築に最適化)
- **Backend/API**: Firebase (BaaSとして完全統合)
- **Database**: Firestore
- **Authentication**: Firebase Authentication (Googleログイン)
- **Hosting**: Netlify または Firebase Hosting

## 4. データ構造と処理フロー

### データベース設計（Firestore）
プロのアーキテクチャ見地から、トップや履歴画面のクエリ負荷を下げるため1つの `expenses` コレクションに統合します。

| コレクション | フィールド名 | データ型 | 説明 |
| :--- | :--- | :--- | :--- |
| `users/{uid}/expenses` | `amount` | number | 支出金額 |
| | `category` | string | `"food"` または `"transport"` |
| | `date` | timestamp| 記録日時 |
| | `memo` | string | 自炊、外食、チャージなどのテキストテキスト |

## 5. セキュリティ・リーガル宣誓

> [!CAUTION]
> 実装担当は以下のルールを **絶対遵守** して進めること。

### セキュリティ対策（必須）
- **機密情報の保護**: APIキー等はソースコードに直接書かず、必ず `.env` を利用して隠蔽し `.gitignore` で除外すること。
- **データアクセス保護**: Firestore Security Rulesを設定し、`request.auth.uid == userId` の条件で他人のデータに絶対アクセスできないよう制限をかけること。
- **デプロイ前厳格監査**: 本番環境へデプロイする前に、**「ワークスペースの徹底分析」**と**「`.gitignore` / `.env` の最新化」**を実施し、機密情報を保護すること。

### 法務対応
- Google認証を利用しデータを保持するため、簡易的なプライバシーポリシー（第三者提供：Firebaseの利用明記、データ削除の仕組み）を準備しておくこと。

## 6. 実装ステップと受け入れテスト

> ※手戻りを防ぐため、以下の順序を厳守し、各Stepごとに「受け入れテスト」をクリアしてから次に進むこと。

### Step 1: 環境構築とベースUI
- **やること**: Flutterの初期化、グラスモーフィズム・丸文字フォント・淡いトーンのテーマ設定。トップ画面のColumn/Rowレイアウト。織璃無画像の静的配置。
- **受け入れテスト**: スマホサイズのブラウザで、下部ナビゲーションと大枠のレイアウトが正しく表示されること。

### Step 2: 認証・DB設定 (Firebase)
- **やること**: Firebase接続、Googleログイン機能、Firestore初期セキュリティルール。
- **受け入れテスト**: ログインボタンからGoogleアカウントのUIDを取得できること。

### Step 3: 手動入力フォームとHome連携（Must）
- **やること**: 記録入力画面（TabBar、電卓/キーボードUI、スナックバー）。Firestoreへの書き込み処理。トップ画面の合計値の計算・表示。
- **受け入れテスト**: 電卓で金額を入力し保存すると、スナックバーが表示され、トップ画面の「月の合計金額」が増加していること。

### Step 4: 振り返りリスト画面（Must）
- **やること**: Firestoreからの当月データ取得（状態管理）。ListViewでの日付降順表示。織璃無の固定テキスト表示。
- **💥 受け入れテスト（正常系）**: Step 3で追加したデータが、振り返り画面のリストに正しく降順で表示されること。
- **💥 受け入れテスト（異常系）**: 未入力のまま保存ボタンを押した際、「金額が未入力です」とエラーハンドリングされること。

### Step 5: デプロイ準備と監査 (Audit)
- **やること**: 不要ファイルを整理し、`/audit` ワークフローを実行。Netlify または Firebase Hosting へデプロイ。
- **受け入れテスト**: 本番環境URLで一連のDay 1動作がエラーなく完走すること。
